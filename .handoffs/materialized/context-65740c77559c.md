## Decisions
- Keep the route as a thin loader and push layout logic into TSX components
  - why: The frontend will iterate faster if page.tsx mostly handles data fetching and leaves composition to React components.
  - from: fe_workspace_shell_001
  - files: dashboard/src/app/page.tsx, dashboard/src/components/dashboard.tsx
- Use bands and tabs instead of a single scrolling report
  - why: Operators need to switch between live concerns quickly without losing where they were in the interface.
  - from: fe_workspace_shell_001
  - files: dashboard/src/components/dashboard.tsx, dashboard/src/components/ui/tabs.tsx
- Surface activity inside the main shell instead of a separate page
  - why: Context switching between pages would slow triage when the operator only wants to inspect what moved recently.
  - from: fe_activity_rail_002
  - files: dashboard/src/components/dashboard.tsx
- Derive activity from session metadata instead of introducing a second event log
  - why: The repo already has enough signal in updatedAt, status, and touched files to bootstrap the rail without a new backend.
  - from: fe_activity_rail_002
  - files: dashboard/src/lib/data.ts, dashboard/src/app/api/data/route.ts
- Use inline expansion rather than modal drill-down for session details
  - why: Inspecting many sessions in sequence is faster if the operator can stay in one scroll context.
  - from: fe_session_details_003
  - files: dashboard/src/components/dashboard.tsx
- Lead with recorded rationale before low-level file counts
  - why: The whole point of the handoff repo is preserving intent; the UI should reinforce that priority.
  - from: fe_session_details_003
  - files: dashboard/src/components/dashboard.tsx
